Error Statuses and Severity Levels
An interface message can have one or more errors. Each error has a status associated with it, as described below. An error’s status is displayed in the Errors table on the Message Details screen. It is important to note that only errors with a status of Active are aggregated and displayed on the Error Summary.
•Active: The error is currently active and unresolved.
•Ignored: A user has determined that the error can be safely ignored and has indicated this by changing its status to Ignored in the Data Management Application (see Marking Errors as Ignored).
Each error has a severity level associated with it, described below. The severity level of an error is displayed on the Error Summary, and also on the Errors table on the Message Details screen.
•Warning: The error is not severe enough to cause the message to fail processing. As long as there are no other errors on the message with a severity of Failure, then the message will be successfully processed (the Message Details will show that the message has a status of Success, and that there is an error with a severity of Warning).
EXAMPLE:
An error due to a missing code mapping for a non-required field would have a severity of Warning.
•Failure: The error is severe enough to cause the message to fail processing. The Message Details will show that the interface message has a status of Failed, and that there is an error with a severity of Failure. Although the specific message is not processed, the interface does move on to the next message.
EXAMPLE:
An error due to a missing code mapping for a required field would have a severity of Failure.
•Critical: The error is severe enough to have stopped the interface from processing messages.
EXAMPLE:
An error due to a incorrect IP address for the destination system would have a severity of Critical.
•Undefined: An unknown or edge case error that has stopped the interface from processing messages and will require intervention by Commure support.
